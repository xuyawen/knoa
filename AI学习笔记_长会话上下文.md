## 一、滑动窗口 Sliding Window

### 1. 这是什么 / 解决什么问题
LLM 有上下文窗口上限（几 K ~ 几十万 token 不等）。对话一长，把所有历史全塞进 prompt 会：① 超窗口直接报错；② 没超也越塞越贵、越塞越慢；③ 无关早期内容稀释注意力。
滑动窗口是最朴素的解法：**只保留最近 N 条消息（或最近 N token），更早的直接丢弃**。窗口之外的内容对当前这一轮"不可见"。

本项目已经落地的就是这一种——`_load_session_history` 拉最近 20 条拼进 `st.messages`。

**场景对比：无窗口 vs 有窗口**
- **无窗口（把所有历史都喂）**：用户连聊 200 轮，每轮都重发 200 轮全文，token 成本线性爆炸；且多数模型窗口根本装不下，直接 400。
- **有窗口（最近 20 条）**：每轮只发最近 20 条，成本可控、不超窗。代价是 20 条之前聊的（"这产品是无线充电器"）滚出去后，模型就看不到了——这正是后续"滚动摘要"要补的。

### 2. 核心代码（本项目已落地，逐行注释）—— `app/core/rag/agent.py`
```python
# 窗口大小：最近多少条历史消息注入 LLM（约 10 轮对话）
MAX_HISTORY_MESSAGES = 20

async def _load_session_history(self, session_id: uuid.UUID) -> list[dict]:
    # 从 DB 按时间正序捞出该会话的全部消息
    result = await self.db.execute(
        select(ChatMessage)
        .where(ChatMessage.session_id == session_id)
        .order_by(ChatMessage.created_at.asc())
    )
    all_msgs = result.scalars().all()

    # 排除最后一条：那就是本轮刚 flush 的 user 消息，后面会单独追加，不能重复
    if len(all_msgs) > 1:
        all_msgs = all_msgs[:-1]
    else:
        return []                       # 第一条消息，没有历史可加载

    # 截断到窗口大小：只留最后 20 条，更早的丢弃
    if len(all_msgs) > self.MAX_HISTORY_MESSAGES:
        all_msgs = all_msgs[-self.MAX_HISTORY_MESSAGES:]

    history: list[dict] = []
    for msg in all_msgs:
        if msg.role == "user":
            # 用户消息若带附件（图片等），回构成多模态 content blocks
            if msg.attachments:
                content = self._build_user_content(msg.content or "", msg.attachments)
            else:
                content = msg.content or ""
            history.append({"role": "user", "content": content})
        elif msg.role == "assistant":
            if msg.content:
                history.append({"role": "assistant", "content": msg.content})
    return history
```

注入位置（`stream_answer` 里构造 `st.messages`）：
```python
# 先加载历史，再拼成 [system, *history, 当前用户消息]
history_msgs = await self._load_session_history(session.id)
st.messages = [
    {"role": "system", "content": AGENT_SYSTEM_PROMPT + self._memory_section()},
    *history_msgs,
    {"role": "user", "content": user_content},
]
```

### 3. 为什么这样做（场景对比）
**场景一：用户问"这个产品需要什么认证"，前 5 轮刚识别过它是无线充电器**
- 修之前（不加载历史）：LLM 看到的只有 `[system, "这个产品需要什么认证"]`，完全不知道"这个产品"是谁，只能回"请补充产品名称"。
- 修之后（滑动窗口）：`[system, …历史 5 轮…, "这个产品需要什么认证"]`，模型顺着上下文知道指的是无线充电器，正常作答。

**场景二：窗口定多大**
- 定太大（比如 200 条）：每轮重发巨量历史，token 贵、延迟高，且早期寒暄/无关内容稀释注意力。
- 定太小（比如 2 条）：刚聊的设定下一轮就丢了，体验像"金鱼记忆"。
- 经验值：按 token 算比按"条数"算更稳（一条用户消息可能 10 字也可能 2000 字）。本项目用"20 条"是简化版，严格应按 token 预算截断（见第九节落地建议）。

### 4. 面试官视角提问
**Q1（基础）：** 长会话为什么不能直接把所有历史都塞进 prompt？
> 答点：① 模型有上下文窗口硬上限，超了直接报错；② 没超也每轮线性涨成本、涨延迟；③ 无关早期内容稀释注意力（lost-in-the-middle）。所以必须做上下文管理。

**Q2（进阶）：** 滑动窗口是最简单的做法，它最大的缺陷是什么？
> 答点：FIFO 丢弃——窗口外的早期内容永久不可见。用户 50 轮前定过的"就用英文回复"，一旦滚出窗口模型就忘了。所以它只是打底，长会话还得配摘要/长期记忆才能不丢关键信息。

**Q3（追问·本项目）：** 你这个历史加载是按"条数"截断的，按 token 截有什么好处？
> 答点：一条消息长度方差极大（"你好" vs 一段 2000 字文档），按条数截断会导致"轮数相同但 token 量天差地别"，成本不可控、还可能偶发超窗。按 token 预算（如 ~4K）截断，成本和安全性都更稳，且能精确控制 system + 历史 + 检索结果的总占用。

## 二、滚动摘要 Rolling Summary

### 1. 这是什么 / 解决什么问题
滑动窗口的死穴是"窗口外的内容彻底没了"。滚动摘要补这一刀：**维护一段"对话摘要"文本，每当旧消息滚出窗口，就调一次 LLM 把它压进摘要，然后丢弃原始消息**。下一轮给模型的变成 `[system, 摘要, 最近N条]`。
这样早期语义被保留成"压缩后的记忆"，而不是凭空消失。ChatGPT 那类产品的"它记得我们之前聊过什么"底层多半是这机制（或类似的长期记忆）。

**场景对比：纯窗口 vs 窗口 + 滚动摘要**
- **纯窗口（20 条）**：用户前 30 轮定过"我是做 3C 配件跨境的，主推无线充电器"，第 31 轮起这设定滚出窗口，之后模型答"你们卖什么品类"就瞎编。
- **窗口 + 摘要**：那 30 轮的精华（"3C 跨境 / 主推无线充电器"）被压进一段 summary，之后每轮都带着，模型始终记得用户画像。

### 2. 核心思路（业界做法·示意）
```python
# 当历史消息超出窗口阈值，把"窗口外的旧消息"交给 LLM 压缩成摘要，存回会话
async def _roll_summary(self, session_id, old_msgs, llm):
    # old_msgs = 即将被窗口挤掉的那批早期消息
    prev_summary = await self._get_summary(session_id)      # 上一次的摘要（可能为空）
    prompt = (
        f"已有摘要：{prev_summary}\n"
        f"新增对话：{_msgs_to_text(old_msgs)}\n"
        "请把以上合并成一段简洁的对话摘要，保留用户画像、已确认的事实、待办，"
        "丢弃寒暄和已解决的小事。"
    )
    new_summary = await llm.chat([{"role": "user", "content": prompt}])
    await self._save_summary(session_id, new_summary)     # 写回 ChatSession.summary 之类的字段
    return new_summary
```
注入时把摘要放在 system 之后、近期消息之前：
```python
st.messages = [
    {"role": "system", "content": SYSTEM + summary_section},
    *recent_window,          # 最近 N 条原始消息
    {"role": "user", "content": user_content},
]
```

### 3. 为什么这样做（场景对比）
**场景：超长多轮、且早期有"人设/事实"设定**
- 不做摘要：设定滚出窗口即失忆，长对话体验崩。
- 做滚动摘要：设定被压缩保留，且摘要本身随对话"滚动更新"，越聊越精炼，不占太多 token。

**代价与坑**
- 每次滚动要花一次 LLM 调用（可异步、可攒批，不让用户等）。
- 摘要本身可能"丢细节"或"自己编造"——所以摘要一般只承"人设/事实/待办"，不承"逐字依据"；真要溯源还得回 DB 查原始消息。

### 4. 面试官视角提问
**Q1（基础）：** 滚动摘要和"把所有历史塞进 prompt"本质区别？它解决了窗口的什么短板？
> 答点：窗口是 FIFO 硬丢，摘要是把将要丢失的内容"语义压缩"成一小段保留。短板就是"窗口外永久失忆"，摘要让早期关键信息以压缩形态续命。

**Q2（进阶）：** 摘要存在哪里？和原始消息怎么分工？
> 答点：摘要存会话级字段（如 `ChatSession.summary`），原始消息仍存 `ChatMessage` 表（溯源/审计用）。prompt 里给模型的是"摘要 + 最近窗口原始消息"，既不丢人设、又能看到近期原话。

**Q3（追问）：** 每次滚动都调一次 LLM，成本怎么控？
> 答点：① 异步化——滚动在后台任务跑，不阻塞当前回答；② 攒批——不是每一条滚出都调，而是攒到"超出阈值一段"才压一次；③ 只对长会话触发，短对话根本走不到滚动逻辑。

## 三、压缩 Compression

### 1. 这是什么 / 解决什么问题
摘要偏向"生成自然语言小结"。压缩是更广义、有时更激进的做法：目标是**用更少 token 表达同样的信息**，手段不止摘要一种：
- 把多轮"用户问 + 助手答"合并成一条紧凑陈述
- 删除低价值消息（"好的""收到"、工具调用的中间脚手架）
- 对长消息做截断/抽关键句
本质是"有损但可控"地缩减，和摘要互补——摘要重"语义保真"，压缩重"体积压减"。

**场景对比：只摘要 vs 摘要 + 压缩**
- 只摘要：摘要段会随对话越滚越长，最终自己也变成"长上下文"的负担。
- 摘要 + 压缩：摘要段本身也限长、超长就再压一层，形成"摘要也有预算"的闭环。

### 2. 核心思路（业界做法·示意）
```python
def _compress_messages(msgs, budget_tokens: int) -> list[dict]:
    # 1) 先丢明确无价值的消息（寒暄、纯确认、工具中间态）
    kept = [m for m in msgs if not _is_low_value(m)]
    # 2) 若仍超预算，把相邻的用户/助手对合并成一条紧凑消息
    if _est_tokens(kept) > budget_tokens:
        kept = _merge_pairs(kept)
    # 3) 仍超则截断单条过长消息（保留首尾，中间省略）
    return _truncate_long(kept, budget_tokens)
```

### 3. 为什么这样做（场景对比）
**场景：一段对话里夹了大量"工具调用过程"和寒暄**
- 不压缩：这些占 token、且对当前问题无增益，纯浪费 + 稀释注意力。
- 压缩：寒暄/工具脚手架直接丢，把"问 A → 答 A"压成一条，关键信息密度高，模型和钱包都轻松。

### 4. 面试官视角提问
**Q1（基础）：** 压缩和摘要听起来像一回事，区别在哪？
> 答点：摘要=生成一段自然语言小结（语义保真优先）；压缩=更广义的体积压减，含删低价值消息、合并相邻轮次、截断长消息等，不一定生成新文本，目标是"更少 token 表达同信息"。两者常组合用。

**Q2（进阶）：** 压缩是有损的，怎么避免"压掉关键信息"？
> 答点：① 只丢明确低价值的（寒暄/确认/工具中间态），绝不丢用户指令和事实性回答；② 合并/截断优先动"中间过程"而非"结论"；③ 关键事实优先走长期记忆（见第六/七节）兜底，即使压缩误删也能从记忆召回。

## 四、重要性裁剪 Pruning

### 1. 这是什么 / 解决什么问题
滑动窗口是"按位置丢"（最旧的先走）。但最旧的不一定最不重要——可能第 3 轮就定下了"用户是做 3C 的"这个人设，第 50 轮才是无关寒暄。
重要性裁剪改成"按价值丢"：给每条消息打分（是否含用户指令、是否含关键事实、是否被广泛引用），优先保留高分、丢弃低分。

**场景对比：FIFO 窗口 vs 重要性裁剪**
- **FIFO**：人设设定在第 3 轮，第 23 轮被窗口挤掉，第 50 轮模型已失忆。
- **重要性裁剪**：人设消息得分高，即使很旧也留在窗口；第 50 轮的寒暄得分低，先被丢。

### 2. 核心思路（业界做法·示意）
```python
def _score(msg) -> float:
    s = 0.0
    if msg.role == "user":               # 用户消息权重高（指令来自用户）
        s += 2.0
    if _has_fact_or_preference(msg):    # 含"我是做 X 的""用英文回复"等事实/偏好
        s += 3.0
    if _is_referenced_later(msg):       # 后续消息引用过它（如"按刚才说的"）
        s += 1.5
    if _is_greeting_or_ack(msg):      # 寒暄/确认类
        s -= 1.0
    return s

def _prune(msgs, keep: int) -> list[dict]:
    # 按分数从高到低保留 keep 条，而非按时间
    return sorted(msgs, key=_score, reverse=True)[:keep]
```

### 3. 为什么这样做（场景对比）
**场景：长对话里早期有人设、后期多寒暄**
- FIFO：早期人设先丢，体验崩。
- 重要性裁剪：人设留、寒暄走，模型"记得重要的事、忘了废话"。

### 4. 面试官视角提问
**Q1（基础）：** 重要性裁剪相比 FIFO 窗口好在哪？评分看哪些信号？
> 答点：FIFO 按位置丢，最旧≠最不重要；裁剪按价值丢，人设/事实/被引用的消息高分保留，寒暄低分先走。信号：是否用户指令、是否含事实/偏好、是否被后续引用、是否纯寒暄确认。

**Q2（进阶）：** 打分本身也要成本，怎么平衡"算分开销"和"裁剪收益"？
> 答点：① 不必每条都调 LLM 打分——可用规则（含"我是/请用/记住"等关键词加权）+ 轻量 heuristic；② 只在窗口真满、必须丢时才触发打分，不满不计算；③ 打分结果可缓存到消息上，避免每轮重算。

## 五、检索式历史 Conversation-as-RAG

### 1. 这是什么 / 解决什么问题
前面几种都是"把历史塞进 prompt"（不管窗口/摘要/压缩，终究还是往上下文里灌）。检索式历史换了个思路：**不塞全部历史，而是把过往消息做向量化存起来，每轮只召回"和当前问题相关"的那几条**。等于把对话当成知识库来检索。

**场景对比：灌历史 vs 检索式历史**
- **灌历史（窗口/摘要）**：长对话里 90% 历史对当前问题无关，却仍占 token。
- **检索式历史**：用户现在问"无线充电器的认证"，只召回之前聊过"无线充电器""认证"的那几条，其余不进 prompt，又准又省。

### 2. 核心思路（业界做法·示意）
```python
# 每轮问答后，把这条 (用户问 + 助手答) 向量化存进"对话记忆库"
await conv_store.add(session_id, text=f"Q: {q}\nA: {a}", embedding=emb)

# 下一轮提问时，只召回和当前问题最相关的历史片段
related = await conv_store.search(session_id, query=current_question, top_k=5)
history_ctx = "\n".join(r.text for r in related)   # 只这些进 prompt
```

### 3. 为什么这样做（场景对比）
**场景：超长、跨很多话题的会话**
- 灌历史：要么窗口太小丢关键信息，要么窗口太大 token 爆炸。
- 检索式：每轮按需取相关片段，话题再多也不爆窗，且"跨轮召回"比 FIFO 更聪明。

**代价**
- 实现复杂：要维护一个对话向量的写入/检索通道（可复用现有 RAG 的向量库，见核心功能笔记第二节）。
- 可能漏掉"相关但没被召回"的上下文——所以常和"近期窗口"组合：近期几条必带 + 更早的按需召回。

### 4. 面试官视角提问
**Q1（基础）：** 检索式历史和前面"把历史塞进 prompt"的根本区别？
> 答点：前面几种终归是往上下文灌（窗口限量、摘要压缩、裁剪选优，但都是"全量进 prompt 的子集"）；检索式是"按需召回"——历史向量化存库，每轮只取和当前问题相关的几条，不相关的根本不进 prompt。

**Q2（进阶）：** 它有什么固有风险？怎么缓解？
> 答点：召回可能漏掉"相关但语义不像"的上下文（比如用户用不同说法提到同一件事）。缓解：和"近期 N 条必带"组合——近期原话保底、更早的按需召回，两者并集进 prompt。

## 六、分层记忆 Tiered Memory

### 1. 这是什么 / 解决什么问题
仿人脑记忆分层，是最成体系的 Agent 上下文方案：
- **工作记忆（Working）**：最近几条消息，必进 prompt（快、准、即时）。
- **短期记忆（Short-term）**：滚动摘要承载，压缩后的近期语义。
- **长期记忆（Long-term）**：抽取的**关键事实/用户偏好**，跨会话、跨长对话都生效（本项目 Mem0 就属于这一层）。

三层各管一摊，组合后就是"即时上下文 + 近期压缩 + 跨会话事实"的完整方案。

**场景对比：单层窗口 vs 分层记忆**
- **单层窗口**：人设滚出窗口即失忆，且跨会话（刷新/新对话）彻底归零。
- **分层记忆**：人设走长期记忆，刷新页面、开新对话都记得；近期对话走窗口+摘要；当前轮走工作记忆。

### 2. 与本项目 Mem0 的关系（已落地部分）
本项目的 `MemoryStore`（核心功能笔记第七节）就是**长期记忆层**：
- 抽取：每轮问答后 LLM 从对话里挑"值得长期记"的（偏好/事实/反馈）
- 召回：下一轮按 `user_id` 语义召回 top_k，注入 system prompt
- 去重：余弦 > 0.92 判为同一条 → UPDATE 覆盖，避免矛盾记忆堆积

它管"跨会话事实"，但**不**管"对话流压缩"——那是短期记忆（滚动摘要）的活。两者正交、互补。

### 3. 为什么这样做（场景对比）
**场景：用户第一轮说"用中文、别太啰嗦"，第 40 轮开新对话继续问**
- 只有窗口/摘要：新对话是空上下文，偏好全丢，模型又答得又长又英文。
- 分层 + 长期记忆（Mem0）：偏好存在长期层，新对话召回后注入，模型自然照着来。

### 4. 面试官视角提问
**Q1（基础）：** 分层记忆一般分几层？各层存什么？
> 答点：工作记忆（最近几条，必进 prompt）、短期记忆（滚动摘要，压缩后的近期语义）、长期记忆（抽取的关键事实/用户偏好，跨会话生效）。三层分别解决"即时性/近期性/持久性"。

**Q2（进阶）：** 你们项目的 Mem0 长期记忆属于哪一层？它和"滚动摘要"什么关系？
> 答点：Mem0 是长期记忆层，管跨会话的用户偏好/事实；滚动摘要是短期记忆层，管单会话内的对话流压缩。两者正交——Mem0 不压缩对话、摘要不跨会话，合起来才是完整上下文方案。

## 七、Prompt Caching（成本侧优化）

### 1. 这是什么 / 解决什么问题
严格说它不是"管理上下文内容"，而是"降本"——但凡长会话每轮都重发一大段（system + 历史 + 摘要），成本就高。主流模型（Claude / Gemini / 部分 OpenAI 兼容端点）支持 **Prompt Caching**：把不变的"前缀"（system 指令、长期记忆段、甚至稳定历史）缓存下来，重复发送只收一次钱，后续命中缓存的 token 单价大幅降低。

**场景对比：不缓存 vs 缓存**
- **不缓存**：每轮重发 4K token 的 system+历史，100 轮问答 = 400K token 全按"输入价"计费。
- **缓存**：前缀命中缓存后，那 4K 按"缓存命中价"（常是输入的 1/10）计，长会话成本陡降。

### 2. 核心思路（业界做法·示意）
```python
# 把"稳定不变的前缀"放在 messages 最前面，且保持内容稳定（不要每轮改 system）
# 这样提供商才能把它识别为可缓存前缀
messages = [
    {"role": "system", "content": STABLE_SYSTEM + stable_memory_section},  # ← 这段稳定，进缓存
    *history_or_summary,        # 这段每轮变，不进缓存
    {"role": "user", "content": current_q},
]
# 调用时声明缓存（以 Anthropic 为例）
resp = await client.messages.create(
    model=..., messages=messages,
    system=[{"type": "text", "text": STABLE_SYSTEM, "cache_control": {"type": "ephemeral"}}],
)
```

### 3. 为什么这样做（场景对比）
**场景：长会话每天上万轮问答**
- 不缓存：前缀 token 反复计费，账单爆炸。
- 缓存：稳定前缀只首次全价，后续命中缓存价，成本可降一个数量级。

**关键前提**
- 缓存靠"前缀稳定"命中——如果你的 system prompt 每轮都动态拼不同内容，缓存就失效。所以"稳定指令"和"易变历史"要分开摆放（稳定在前、易变在后）。

### 4. 面试官视角提问
**Q1（基础）：** Prompt Caching 解决长会话的什么问题？它管的是"内容"还是"成本"？
> 答点：管成本，不管内容。长会话每轮重发大段前缀（system+历史），不缓存就反复全价计费；缓存后稳定前缀只首次全价、后续命中缓存价（常 1/10），成本陡降。

**Q2（进阶）：** 想让缓存命中率高，messages 该怎么组织？
> 答点：缓存按"前缀"命中，所以把**稳定不变的内容**（system 指令、长期记忆段）放在最前面且保持内容固定；把**每轮易变的历史/当前问题**放在后面。若 system 每轮都动态拼不同串，前缀就变了，缓存失效。

## 八、大窗口模型（不是银弹）

### 1. 这是什么 / 解决什么问题
Gemini 1M、Claude 200K 这类"超长上下文窗口"模型出来后，很多人觉得"上下文管理不用做了，全塞进去就行"。窗口大了确实能兜住更长对话，但它**不是银弹**。

**场景对比：小窗口硬塞 vs 大窗口全塞**
- **小窗口（如 4K）硬塞长对话**：超窗直接报错，必须管理。
- **大窗口全塞**：能装下，但① 贵（输入 token 按量计费，长对话每轮都重发巨量历史）；② 慢（prefill 越长延迟越高）；③ **lost-in-the-middle**——研究证明，模型对超长上下文"中间部分"的注意力明显下降，关键信息放中间反而答不准。

### 2. 为什么大窗口不能取代管理
- **成本**：窗口大 ≠ 免费，长上下文每轮重发都是真金白银。
- **lost-in-the-middle**：即使装得下，模型也不一定"注意到"中间的关键设定。所以大窗口通常和新摘要/检索**组合**用——用大窗口兜住"近期一大段"，仍对更早期做摘要/检索。
- **延迟**：prefill 百万 token 的首次延迟不可忽视，实时问答体验受损。

### 3. 面试官视角提问
**Q1（基础）：** 现在模型窗口都上百万 token 了，还需要做上下文管理吗？
> 答点：还需要。三个理由：① 成本——窗口大不等于免费，长对话每轮重发历史都是计费 token；② 延迟——prefill 越长首次响应越慢；③ lost-in-the-middle，超长上下文中间部分的注意力下降，关键信息放中间反而答不准。大窗口通常和摘要/检索组合，而非取代。

**Q2（进阶）：** 什么是 lost-in-the-middle？怎么缓解？
> 答点：指模型对超长 prompt 中间段落的利用率显著低于首尾的现象（NIAH 这类测试证实）。缓解：① 关键信息放在 prompt 首尾；② 控制上下文总量（摘要/裁剪/检索），不无脑全塞；③ 重要事实走长期记忆（独立召回），不依赖它恰好落在窗口好位置。

## 九、我们系统的位置 + 落地路线

### 1. 现在在哪
- ✅ **滑动窗口**：`_load_session_history` 按 20 条截断加载历史（`d1db0fb` 落地，修了"无上下文"的 bug）
- ✅ **长期记忆（Mem0）**：每轮抽取用户偏好/事实，跨会话召回（核心功能笔记第七节）
- ❌ **滚动摘要**：没有。所以对话一长，20 条之前聊的（如"这是无线充电器"）滚出窗口后又会看不见
- ❌ **压缩 / 重要性裁剪 / 检索式历史**：均未做
- ❌ **Prompt Caching**：接的 Agnes 端点是否支持待确认，未启用

### 2. 落地建议（按性价比排）
1. **窗口按 token 算**：把 `MAX_HISTORY_MESSAGES=20` 改成"约 4K token 预算"截断，比按条数稳（一条消息长度方差大）。
2. **加滚动摘要**：会话超阈值时，异步调 LLM 把早期消息压成 `ChatSession.summary` 注入 system；这是补"长对话失忆"最直接的一刀。
3. **开 Prompt Caching**：确认 Agnes 端点支持后启用，长会话降本。
4. **真要做超长多话题**：再上"检索式历史"（复用现有向量库，把对话当 RAG 检索）。

### 3. 面试官怎么讲这块
- 被问"长会话你怎么处理"：先说我们落地了**滑动窗口 + 长期记忆（Mem0）**，再主动展开"业界还有滚动摘要/压缩/检索式历史/分层记忆几种，各自解决窗口的什么短板"，展示你有体系认知而不只会调自己那一种。
- 被追问"你们窗口外的内容丢了怎么办"：诚实说"目前靠 Mem0 长期记忆兜关键事实，对话流本身的早期内容在超长时会丢，这是我们下一步要补滚动摘要的点"——比硬吹"我们全做了"更可信。

## 十、面试节奏建议（怎么用上面这些）
1. **被问"长上下文你怎么管"**：按"窗口（打底）→ 摘要（补失忆）→ 长期记忆（跨会话）→ 检索式（按需召回）→ 大窗口模型的局限"的顺序讲。这是"由浅入深、层层加难"的叙事线，面试官会被你带着走。
2. **被追问细节时**：一（20 条窗口的代码）、二（滚动摘要的"合并摘要+丢弃原始"机制）、六（分层记忆 vs 我们 Mem0 的对应关系）、八（lost-in-the-middle）是四个最容易出彩的点，提前背熟。
3. **不会的别硬编**：比如"检索式历史"我们没做，可以说"思路是复用现有向量库把对话当 RAG 检索，近期窗口保底 + 更早按需召回，目前还没落地但架构上直接复用第二节的混合检索"——展示你有架构视野但不吹已完成。
